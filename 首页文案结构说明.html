怎么读这份文案

这份文档把 ModelGo 官网首页的文案，按「区块 → 层级 → 每块讲什么」理清楚，方便你排版。 重点看「平台能力」那一段——它由 4 个结构完全一致的区块组成，之前给你的稿子层级混乱、还漏了内容，以这份为准。

通用排版规则（先记这条）
  • · 平台能力每个区块结构都一样：英文眉标 + 中文主标题 + 一句总述 + 3 张特性卡（卡 = 小标题 + 一句描述）。四块只是内容不同。
  • · 层级只有 3 级：① 英文眉标（如 BILLING）② 主标题 + 总述 ③ 特性卡。别再往下细分。
  • · 数字、图表是动画/组件，不是文案。比如 OBSERVABILITY 里的滚动数字（odometer 效果），别当正文排进去。
  • · 英文眉标（BILLING / OBSERVABILITY…）是小号装饰标，中文主标题才是视觉重点。

04

平台能力 （重点 · 4 个区块）

区块英文标：PLATFORM FACTS

结构公式（4 块通用）：英文眉标 + 中文主标题 + 一句总述 + 3 张特性卡（标题 + 描述）
BILLING

Token 级计费，预算级掌控

每一次调用按输入、输出、缓存逐项计账，配合三层预算熔断与多维成本拆解，企业 AI 开支笔笔有据。

毫秒级 Token 流水

输入、输出、上下文缓存逐项分账，每笔调用可追溯、逐笔可对账

三层预算熔断

工作空间、成员、API Key 各设硬性上限，触顶即刻拦截，没人能打穿共享余额

多维消费穿透

按工作空间、成员、模型、Key 逐层下钻，成本分摊与用量趋势一目了然

OBSERVABILITY

全链路调用日志，三维用量看板

消费、请求、Token 三维看板配毫秒级调用日志，按模型、密钥、状态码任意下钻；OpenTelemetry 标准协议，无缝融入企业既有观测体系。

🎨 注意：主视觉里那组一直跳动的数字是滚动数字动画（odometer），是组件效果、不是文案——之前稿里那串 98/99/01… 就是它，别排进正文。
三维用量看板

消费、请求、Token 同屏可视化，趋势与调用规律一眼看清

毫秒级日志下钻

按模型、状态码、密钥、Request ID 细分检索，单条日志内看全状态、延迟与护栏命中原因

开放观测标准

兼容 OpenTelemetry，一键路由 Datadog、Grafana，不打断你现有的观测体系

MULTI-TENANCY

为成长型团队打造的身份隔离网格

从单打独斗到多部门协作，「租户 - 工作空间 - 成员」三级身份矩阵，配合角色授权与全程管理审计，建空间、分权、加人平滑扩展，无需推倒重来。

🎨 注意：这块也是 3 张卡（之前给你的稿只写了 1 张，是漏的）。
三级身份矩阵

租户管账务、工作空间管运行时、成员管归因，边界清晰，扩张不用重构

最小权限授权

Owner / Admin / Member 角色分明，成员只在授权空间内操作，只看见自己该看见的 Key 与日志

全程管理审计

建 Key、改护栏、变更成员，每一步管理动作都留痕可回查，交接与追责都有据

SECURITY

让 AI 调用尽情狂奔，而安全始终在轨道内

模型黑名单、IP 白名单与三层预算护栏编织进每一次调用，越界请求在网关层被拦下，命中哪条规则、为什么被拦，日志里直接可查。

🎨 注意：SECURITY 是平台能力的第 4 个区块，之前稿里整块漏了，别忘做。
模型黑名单

自主圈定禁调模型，越权调用在网关层硬拦截，可用范围锁死在企业安全域内

IP 信任域

白名单限定调用来源，只允许受信 IP / CIDR 发起请求，密钥即使泄露也调不通

三层护栏栈

全局、个人、Key 三层策略叠加，更严者生效，任何一层触线即拦、命中原因可查
